# Frontend Module

Files:

- `src/main/webapp/index.html`
- `src/main/webapp/style.css`
- `src/main/webapp/script.js`

The frontend is a vanilla browser application. It has no build step and no framework. Tomcat serves these files directly from the WAR.

## Current Runtime Mode

The cleaned project is configured for the real Java backend by default:

```javascript
const CONFIG = {
  MOCK_MODE: false,
  GET_PRODUCT_URL: "get-product",
  CHECKOUT_URL: "checkout",
  CURRENCY: "\u20b9",
};
```

The actual code displays the Rupee symbol in the browser UI. The important deployment detail is that `GET_PRODUCT_URL` and `CHECKOUT_URL` are context-relative. This lets the same frontend work when the WAR is deployed at:

```text
http://localhost:8080/billing-system/
```

With context-relative paths, `fetch("get-product")` resolves to:

```text
http://localhost:8080/billing-system/get-product
```

That is the correct servlet URL for a WAR named `billing-system.war`.

## Mock Mode

Use mock mode only for standalone frontend testing:

```javascript
MOCK_MODE: true
```

In mock mode:

- Product lookup uses the local `MOCK_PRODUCTS` object.
- Checkout does not call the servlet.
- The payload is logged to the browser console.
- No PDF is generated by the backend.

The mock product UIDs are:

```text
P001 P002 P003 P004 P005 P006
```

Before building for Tomcat deployment, switch back to:

```javascript
MOCK_MODE: false
```

## HTML Structure

`index.html` contains the POS shell:

- Header bar with clock and cashier label.
- Scanner input with `id="uid-input"`.
- Cart table body with `id="js-cart-body"`.
- Summary panel with subtotal, tax, discount, and grand total fields.
- Receipt overlay with a PDF iframe, print button, download link, and close button.

The receipt overlay was cleaned up during the revision. There is now one overlay and one `js-receipt-download` element, avoiding duplicate IDs and event binding ambiguity.

## JavaScript Flow

### Product Scan

```text
User enters UID
  -> fetchProduct(uid)
  -> addOrIncrementItem(product)
  -> renderCart()
  -> recalcAll()
```

`fetchProduct` either reads from `MOCK_PRODUCTS` or calls:

```text
get-product?uid=P001
```

The backend servlet mapping is still `/get-product`; the frontend uses `get-product` so the browser keeps the current application context.

### Cart State

The cart is stored in a `Map`:

```javascript
const cart = new Map();
```

The key is the product UID. The value contains product name, base price, tax rate, quantity, and discount percentage.

The DOM is treated as a rendering target, not as the source of truth. Cart calculations always read from the `Map`.

### Calculation

Per item:

```text
base amount = base price * quantity
discount    = base amount * discount percent / 100
taxable     = base amount - discount
tax         = taxable * tax rate / 100
total       = taxable + tax
```

Grand totals are calculated by summing the per-item values.

### Checkout

When the user clicks Complete Checkout, the frontend POSTs JSON to:

```text
checkout
```

The servlet mapping is `/checkout`, and the context-relative frontend URL resolves to `/billing-system/checkout` after deployment.

The payload shape is:

```json
{
  "subtotal": 900.0,
  "discount": 45.0,
  "taxAmount": 65.56,
  "grandTotal": 920.56,
  "items": [
    {
      "productUid": "P001",
      "productName": "Basmati Rice 5kg",
      "basePrice": 450.0,
      "quantity": 2,
      "taxRate": 5.0,
      "discountPct": 0.0,
      "calculatedTax": 45.0,
      "finalItemPrice": 945.0
    }
  ]
}
```

## Receipt Overlay

When checkout succeeds in real mode, `CheckoutServlet` returns PDF bytes. The frontend converts the response to a `Blob`, creates a temporary object URL, loads it in the iframe, and attaches the same URL to the download link:

```javascript
const pdfBlob = await res.blob();
const blobUrl = URL.createObjectURL(pdfBlob);
showReceiptOverlay(blobUrl);
```

When the overlay closes, the object URL is revoked to free browser memory.

The print button uses the iframe's `contentWindow.print()` method.

## CSS Notes

The UI uses a dark POS-terminal style. Important layout pieces:

- `.layout` uses CSS Grid for the cart and summary columns.
- `.panel` frames the cart and summary areas.
- `.table-wrap` provides scrolling for the cart table.
- `.overlay` and `.overlay__box` handle receipt preview.
- `.btn--accent` styles the print action.

The UI collapses to one column on small screens.
